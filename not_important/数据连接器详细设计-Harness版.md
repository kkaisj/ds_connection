# 数据连接器（DC）详细设计文档（Harness 版）

## 1. 文档目标
本文件使用 Harness Engineering 思维，将“产品方案”落地为可执行工程文档，确保 AI Agent 与研发团队在同一约束下稳定迭代。

目标：
1. 让需求可实现：从功能点转成模块、接口、数据模型和里程碑。
2. 让实现可约束：明确分层边界、依赖方向、编码规范、禁止事项。
3. 让迭代可闭环：通过自动化测试、评估标准和回归机制实现持续交付。

---

## 2. North Star 与范围

### 2.1 North Star
DC 是面向电商多平台的数据连接基础设施：统一接入、统一调度、统一落库、统一监控告警。

### 2.2 MVP 范围（首期）
1. 平台连接市场（应用展示、平台筛选、连接引导）。
2. 连接任务管理（参数配置、调度、执行、重试）。
3. 店铺账号管理（多平台凭据、绑定关系、脱敏展示）。
4. 存储管理（MySQL、飞书多维表、钉钉表格）。
5. 消息通知（飞书/钉钉/企微 Webhook）。
6. 管理后台（应用上架、模板管理、版本发布）。

### 2.3 非目标（MVP 外）
1. 多租户计费系统。
2. 可视化拖拽编排。
3. 外部插件市场开放协议。

## 3. Harness 执行模型

### 3.1 三角色职责
1. Planner：把需求拆为“可验收”的功能规格与 Sprint 目标。
2. Generator：按 Sprint Contract 实现功能与测试。
3. Evaluator：按统一评分标准做验收、回归和驳回。

### 3.2 交付流（每个 Sprint）
1. Planner 产出 `Sprint Contract`（范围、完成定义、验收项）。
2. Generator 完成代码、迁移脚本、测试、文档更新。
3. Evaluator 执行评估：功能、稳定性、边界合规、可观测性。
4. 不达标则驳回并附修复清单；达标进入下一 Sprint。

### 3.3 Done 定义（统一）
1. 功能满足合同中所有验收条目。
2. 单测 + 集成测试通过。
3. 无高风险安全问题（P0/P1）。
4. 关键链路具备日志、指标、告警。
5. 架构边界检查通过（无非法跨层依赖）。

## 4. 系统架构与模块边界

### 4.1 分层架构
1. Presentation 层：FastAPI Router、Vue 页面。
2. Application 层：任务编排、业务服务、用例协调。
3. Domain 层：实体、值对象、领域规则。
4. Infrastructure 层：数据库、浏览器自动化、消息通知、缓存。

### 4.2 依赖方向（强约束）
1. Presentation -> Application。
2. Application -> Domain / Infrastructure（接口）。
3. Infrastructure -> Domain（仅 DTO/Entity 映射，不反向调用 Application）。

禁止：
1. Domain 依赖 Infrastructure。
2. Repository 调用 Service。
3. Feature A 直接依赖 Feature B 的内部实现。

### 4.3 领域模块（建议）
1. `marketplace`：连接市场与应用目录。
2. `account`：店铺账号与凭据管理。
3. `task`：任务实例、调度、执行生命周期。
4. `storage`：多存储配置与数据写入适配。
5. `execution`：执行日志、状态追踪、重试机制。
6. `notification`：告警策略、Webhook 发送。
7. `admin`：应用上架、版本管理、模板配置。
8. `common`：审计、鉴权、加密、基础异常体系。

## 6. 核心数据模型（逻辑）

### 6.1 主要实体
1. `platform`：平台定义（一级/二级分类）。
2. `connector_app`：连接应用定义（版本、参数模板、状态）。
3. `shop_account`：店铺账号（凭据加密、状态、归属平台）。
4. `task_instance`：任务实例（店铺绑定、调度、参数、启停）。
5. `task_run`：任务执行记录（状态、耗时、错误码、重试次数）。
6. `storage_config`：目标存储配置（按类型动态字段）。
7. `notification_config`：通知通道与触发策略。
8. `audit_log`：敏感操作审计（变更人、时间、前后值摘要）。

### 6.2 关键关系
1. `platform` 1:N `connector_app`
2. `shop_account` 1:N `task_instance`
3. `task_instance` 1:N `task_run`
4. `task_instance` N:1 `storage_config`
5. `task_instance` N:1 `notification_config`

### 6.3 数据安全
1. 凭据字段加密存储（KMS 或应用层密钥）。
2. 返回前统一脱敏（仅展示后 3-4 位）。
3. 审计日志不可物理删除（仅归档）。

## 7. 核心流程设计

### 7.1 连接应用流程
1. 用户进入连接市场并筛选应用。
2. 选择应用并绑定店铺账号。
3. 填写任务参数与调度策略。
4. 选择存储配置。
5. 启用任务并触发首次执行验证。

### 7.2 任务执行流程
1. 调度器产生执行触发事件。
2. 执行器拉取任务配置并获取账号凭据。
3. DrissionPage 执行采集脚本。
4. 数据经清洗映射后写入目标存储。
5. 写入 `task_run` 结果并推送通知（若失败或命中规则）。

### 7.3 失败重试流程
1. 判定错误类型（可重试/不可重试）。
2. 可重试错误按指数退避重试（最大 N 次）。
3. 超阈值后标记失败并触发告警。
4. 支持人工一键重跑。

## 8. API 契约（v1 草案）

### 8.1 连接市场
1. `GET /api/v1/apps`：应用列表（支持平台筛选）。
2. `GET /api/v1/apps/{app_id}`：应用详情。

### 8.2 账号管理
1. `POST /api/v1/accounts`：新增店铺账号。
2. `GET /api/v1/accounts`：账号列表。
3. `PATCH /api/v1/accounts/{id}`：更新账号状态与字段。

### 8.3 任务管理
1. `POST /api/v1/tasks`：创建任务实例。
2. `GET /api/v1/tasks`：任务列表与过滤。
3. `POST /api/v1/tasks/{id}/run`：手动触发执行。
4. `POST /api/v1/tasks/{id}/retry`：失败重试。

### 8.4 执行记录
1. `GET /api/v1/task-runs`：执行记录查询。
2. `GET /api/v1/task-runs/{run_id}/logs`：步骤级日志。

统一响应：
```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

## 9. 非功能与质量门禁

### 9.1 SLO（MVP）
1. 任务成功率 >= 95%（7 日滚动）。
2. 失败告警时延 <= 5 分钟。
3. API P95 < 300ms（查询类接口）。
4. 系统可用性 >= 99.5%。

### 9.2 安全基线
1. 全量接口鉴权（RBAC）。
2. 参数校验 + SQL 注入防护。
3. 密钥轮换机制与最小权限访问。
4. 操作审计与异常登录告警。

### 9.3 可观测性
1. 指标：任务量、成功率、失败率、重试次数、平均耗时。
2. 日志：任务级 Trace ID 贯穿。
3. 告警：按平台/任务/店铺维度分组聚合。

## 10. Harness 反馈回路设计

### 10.1 自动化检查流水线
1. 静态检查：Ruff/ESLint/类型检查。
2. 架构约束检查：禁止跨层依赖、禁止模块循环依赖。
3. 单元测试：Service/Domain 覆盖率阈值 >= 80%。
4. 集成测试：任务执行链路与存储写入。
5. E2E：核心页面流程（连接、配置、执行、重试）。

### 10.2 Evaluator 评分卡（发布门禁）
1. 功能完整性（30%）：验收用例通过率。
2. 稳定性（25%）：重试与异常处理表现。
3. 架构合规（20%）：边界与依赖方向。
4. 可维护性（15%）：可读性、测试质量、文档一致性。
5. 可观测性（10%）：日志、指标、告警是否闭环。

发布标准：总分 >= 80，且无单项低于 60。

## 11. 迭代计划（Sprint Contracts）

### Sprint 1：基础骨架
交付：
1. 项目脚手架、分层目录、基础鉴权。
2. `platform`/`connector_app`/`shop_account` 基础 CRUD。
3. 基础日志与错误处理。

验收：
1. API 契约可调通。
2. 数据迁移可重复执行。
3. 基础单测通过。

### Sprint 2：任务闭环
交付：
1. `task_instance` 与调度配置。
2. 手动执行与执行记录落库。
3. 失败重试与状态机。

验收：
1. 单任务可完成执行闭环。
2. 失败时可看到结构化错误。
3. 重试策略符合配置。

### Sprint 3：存储与通知
交付：
1. MySQL/飞书存储适配器。
2. Webhook 通知中心。
3. 告警频控与去重。

验收：
1. 数据可落两类目标存储。
2. 失败触发通知并可追踪。
3. 通知无重复风暴。

### Sprint 4：管理后台与上线准备
交付：
1. 应用上架管理与模板版本。
2. 全链路观测大盘。
3. 上线/回滚 SOP。

验收：
1. 灰度发布演练通过。
2. 关键 SLO 达标。
3. 运维手册可执行。

## 12. 熵管理与长期治理
1. 文档漂移检查：代码改动必须同步影响文档。
2. 依赖治理：每月依赖扫描与升级。
3. 死代码清理：按覆盖率和调用图识别无效路径。
4. 质量债台账：按 P0-P3 排序、每 Sprint 固定清偿配额。

---

## 13. 风险清单与应对
1. 页面结构变化导致采集失效。
应对：平台适配器版本化 + 回归用例快照。

2. 登录风控和验证码阻塞任务。
应对：验证码中转、人工兜底、账号健康度评分。

3. 高峰流量下任务堆积。
应对：任务优先级队列 + 分片执行 + 限流保护。

4. 多存储结构不一致。
应对：统一 Canonical Schema + 映射模板版本控制。
