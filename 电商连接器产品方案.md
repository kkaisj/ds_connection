# 数据连接器产品方案

## 1. 项目背景

数据连接器【data-connection】：负责各类平台网站和业务系统网页的数据采集与同步，只做数据的搬运工。产品支持基于 Chrome 内核浏览器的自动化操作与数据抓取，便于后续通过 DrissionPage 统一操控浏览器执行采集任务。


## 3. 产品能力框架

### 3.1 接入层

负责接入外部系统和平台，包括：

- 网页平台：各类业务后台、SaaS 管理台、运营平台、数据查询页面
- 企业系统：ERP、CRM、OMS、财务系统等网页版业务系统
- 浏览器环境：支持基于 Chrome 内核浏览器的数据抓取与自动化操作



### 3.2 同步引擎层

负责执行各类数据同步任务，包括：

- 定时拉取
- 实时推送【飞书、钉钉等平台】
- 增量同步
- 失败重试
- 幂等控制
- 冲突处理

### 3.3 配置与编排层

用于支持业务灵活配置：

- 字段映射配置
- 业务规则配置
- 同步策略配置
- 流程触发编排
- 告警策略配置

### 3.4 运维监控层

用于保障平台稳定运行：

- 任务日志
- 错误日志
- 告警通知
- 任务状态监控
- 接口调用监控
- 审计日志

## 4. 核心功能模块

- 平台授权与账号管理[pass]
- 连接器管理
- 字段映射配置[pass]
- 同步任务中心
- 自动化采集任务中心
- 浏览器实例管理
- 异常处理中心
- 监控告警中心
- 开放 API 与 Webhook
- 租户与权限管理 [pass]

## 5. 总体技术架构

建议采用“轻量微服务 + 事件驱动”的架构模式，在保证扩展性的同时控制系统复杂度。

### 5.1 架构组成

- API 网关：统一入口、鉴权、路由、限流
- Connector 服务：按平台封装适配逻辑
- 自动化采集服务：负责通过浏览器自动化访问网页、登录站点、抓取页面数据
- 标准数据服务：维护统一对象模型与转换逻辑
- 同步任务服务：负责任务调度、增量同步、补偿处理
- 消息队列：解耦异步任务与事件流转
- 配置中心：保存映射规则、租户配置、同步策略
- 监控日志服务：支持链路追踪、异常分析、告警
- 管理后台：提供配置、监控和问题排查能力

### 5.2 核心设计原则

- 平台差异尽量封装在 Connector Adapter 内部
- API 接入与网页自动化采集能力分层建设，避免抓取逻辑污染核心业务服务
- 浏览器自动化能力统一面向 Chrome 内核浏览器设计，降低采集环境适配成本
- 平台外部模型统一映射为内部标准模型
- 高耗时链路尽量异步化
- 任务从设计上支持幂等、重试、死信和补偿
- 从初期就考虑多租户隔离与安全审计

## 6. 技术选型建议

## 6.1 后端技术栈

推荐：

- Python
- FastAPI
- SQLAlchemy
- Pydantic
- DrissionPage

推荐原因：

- 开发效率高，适合快速构建连接器和开放 API
- FastAPI 原生支持类型校验、异步接口和自动生成 OpenAPI 文档
- 对接第三方平台 API、Webhook 和任务编排时实现成本较低
- 更适合连接器产品早期高迭代、强集成的开发节奏
- DrissionPage 适合需要真实浏览器行为的网页登录、动态页面抓取和自动化采集场景
- 可统一操控 Chrome、Edge 等基于 Chrome 内核的浏览器执行抓取任务

结论：

- 核心平台后端推荐使用 Python + FastAPI
- 数据模型定义建议统一基于 Pydantic，数据库访问层使用 SQLAlchemy

自动化采集框架建议：

- 使用 DrissionPage 作为网页抓取与浏览器自动化框架

适用场景：

- 某些平台未开放 API，或开放接口无法满足业务字段采集需求
- 需要登录后台后抓取列表、明细、统计、附件等页面信息
- 页面数据依赖前端渲染，常规 HTTP 请求难以直接获取
- 需要在 Chrome 内核浏览器环境中完成点击、输入、翻页、下载等自动化操作

使用建议：

- 将 DrissionPage 运行在独立采集 Worker 中，避免阻塞主 API 服务
- 采集任务通过 RabbitMQ 或 Celery 下发，执行结果回写 MySQL
- 浏览器登录态、Cookie、账号信息统一加密管理
- 对采集频率、异常重试和站点封禁风险设置保护策略
- 优先适配 Chrome、Edge 等 Chrome 内核浏览器，统一浏览器版本和运行环境

## 6.2 前端技术栈

推荐：

- Vue 3
- TypeScript
- Naive UI
- Pinia
- Vue Router

推荐原因：

- 适合 ToB 管理后台场景
- Vue 3 组合式 API 适合复杂配置型页面开发
- Naive UI 在后台系统、表单、数据展示场景下比较高效
- TypeScript 能提升字段映射、任务配置等复杂交互的可维护性


## 6.3 数据存储

主数据库推荐：

- MySQL

适用场景：

- 核心业务数据
- 授权信息
- 映射规则
- 同步任务配置
- 审计日志

推荐原因：

- 生态成熟，部署与运维成本较低
- 适合承载租户、授权、任务、日志、映射规则等核心结构化数据
- 团队普遍更容易上手，便于和现有业务系统对接

飞书存储支持：

- 飞书多维表格：适合作为轻量配置台、业务录入台或运营协同数据源
- 飞书云文档 / 表格：适合承载业务侧临时映射规则、运营配置和人工补录数据

推荐使用方式：

- MySQL 作为系统主存储，保证事务性、结构化管理和稳定性
- 飞书作为业务协同存储和轻量运营配置入口，通过连接器同步到 MySQL 或从 MySQL 回写展示
- 对飞书侧数据变更设计同步校验机制，避免人工编辑导致主数据不一致

缓存推荐：

- Redis

适用场景：

- Token 缓存
- 热点数据缓存
- 分布式锁
- 限流计数

## 6.4 消息队列

推荐方案分阶段考虑：

- MVP 阶段：RabbitMQ
- 中长期平台化阶段：Kafka

选择建议：

- RabbitMQ 上手快，适合任务分发、延迟重试、死信队列
- Kafka 更适合高吞吐事件流、可回放消费与平台级数据链路

## 6.5 任务调度

推荐：

- APScheduler
- Celery Beat

适用场景：

- 定时拉单
- 定时页面抓取与数据同步
- 增量补偿
- 定时重试

备选：

- Airflow

结论：

- 如果任务以应用内调度为主，优先使用 APScheduler
- 如果已引入 Celery 处理异步任务，建议统一使用 Celery Beat 做周期调度

## 6.6 API 网关

推荐：

- Nginx
- APISIX

备选：

- Kong

结论：

- MVP 阶段可使用 Nginx 作为反向代理和统一入口
- 中长期如果需要细粒度插件化治理，可升级到 APISIX 或 Kong

## 6.7 安全与鉴权

建议方案：

- 对接平台采用 OAuth 2.0 或平台原生授权机制
- 内部会话和接口鉴权采用 JWT 或统一 SSO
- 对 Access Token、Secret 等敏感信息进行加密存储
- 配置敏感字段脱敏展示
- 全链路保留操作审计日志

## 6.8 可观测性

推荐组合：

- Prometheus + Grafana：监控指标与告警
- ELK / EFK：日志检索与错误分析
- SkyWalking 或 OpenTelemetry：链路追踪

结论：

- 基础可观测体系建议至少覆盖指标、日志、链路三个维度

## 6.9 部署与运维

推荐：

- Docker：容器化部署
- Kubernetes：服务编排和弹性扩缩容
- GitLab CI 或 GitHub Actions：持续集成与发布
- 环境变量 + `.env` / Secret Manager：配置管理

Python 配套建议：

- Uvicorn / Gunicorn：FastAPI 服务运行
- Alembic：数据库版本管理
- Poetry 或 uv：依赖管理

## 7. MVP 建议方案

建议首期版本聚焦最核心的业务闭环，避免一开始做得过重。

### 7.1 MVP 范围

- 接入 2 到 3 个典型网页平台或业务系统
- 支持页面数据、任务记录、统计指标 3 个核心对象
- 提供基础的授权管理、同步任务、日志追踪能力
- 支持异常重试和人工补偿

### 7.2 MVP 推荐技术组合

- 后端：Python + FastAPI
- 自动化采集：DrissionPage
- 前端：Vue 3 + TypeScript + Naive UI
- 数据存储：MySQL + 飞书
- 缓存：Redis
- 消息队列：RabbitMQ
- 调度中心：APScheduler 或 Celery Beat
- 网关：Nginx
- 监控：Prometheus + Grafana + ELK
- 部署：Docker + Kubernetes

### 7.3 选择这套方案的原因

- 开发效率和系统稳定性较平衡
- 适合快速落地首版
- 招聘与维护成本相对可控
- 后续扩展到更复杂的平台能力也较顺畅

## 8. 关键技术难点

- 多平台 API 规范不统一，适配复杂度高
- 多站点页面结构差异大，抓取模板维护成本高
- 部分站点缺少开放 API，需要浏览器自动化抓取补充数据
- 平台接口限流与频控限制
- 增量同步与最终一致性处理
- 重复消息消费与幂等控制
- 失败重试、死信队列和补偿机制设计
- 多租户数据隔离与安全控制
- 授权失效与 Token 自动续期
- 自动化抓取过程中的登录态维护、反爬限制和页面结构变更
- 链路追踪和问题定位复杂

## 9. 路线图建议

### Phase 1：基础连接能力

- 接入核心网页平台和业务系统
- 完成页面数据、任务记录、统计指标的标准化建模
- 建立基础同步链路和监控能力

### Phase 2：业务扩展能力

- 增加附件、审批、导出结果等扩展对象
- 增强异常处理和人工补偿能力
- 引入更灵活的字段映射和规则配置

### Phase 3：平台化能力

- 提供低代码流程编排
- 提供开放 API 和开发者接入能力
- 建设连接器市场和生态合作能力

## 10. 成功指标建议

- 接入平台数量
- 已接入商家数量
- 同步成功率
- 平均同步延迟
- 异常恢复时间
- 人工处理工时下降比例
- 数据处理效率提升比例
- 付费转化率与续费率

## 11. 总结

数据连接器的核心价值，在于通过统一连接、网页自动化抓取和标准化同步能力，解决多网页平台、多系统之间的数据割裂问题。产品方案应聚焦高频业务场景，技术方案应优先保证可落地性、稳定性和扩展性。建议从 MVP 起步，先实现页面数据抓取、任务同步和统一存储的基础闭环，再逐步演进到流程编排和开放平台能力。
