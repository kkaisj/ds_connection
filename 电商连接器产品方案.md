# 电商连接器产品方案（DC）

## 1. 项目背景
数据连接器（Data Connection，简称 DC）用于连接各类电商平台与业务系统，完成网页数据采集、任务调度、数据清洗与存储同步。

当前业务痛点：
1. 多平台数据分散，人工导出和整理成本高。
2. 平台页面结构变化快，采集链路不稳定。
3. 缺少统一任务编排、执行监控与失败告警机制。

DC 的核心定位：
1. 统一连接层：屏蔽平台差异，统一采集流程。
2. 统一任务层：统一配置、调度、执行与重试。
3. 统一存储层：统一数据落地规范与可观测能力。

## 2. 产品目标
### 2.1 业务目标
1. 支持主流电商平台账号接入与任务配置。
2. 将关键业务数据自动同步到指定存储（优先 MySQL/飞书多维表）。
3. 建立标准化运行监控与异常通知机制，降低人工巡检成本。

### 2.2 指标目标（MVP）
1. 首批支持 3-5 个平台接入（如淘宝天猫、京东、拼多多、抖音）。
2. 任务成功率 >= 95%（7 日滚动）。
3. 失败任务 5 分钟内可触发告警。
4. 单任务配置时长 <= 10 分钟。

## 3. 用户角色与场景
1. 运营人员：配置连接器任务，查看执行结果，处理异常。
2. 管理员：管理连接市场、平台模板、全局策略与权限。
3. 开发/实施：维护平台适配器、排查复杂失败问题。

典型场景：
1. 新接入店铺，按模板快速创建采集任务。
2. 每日定时采集订单/商品/库存数据并入库。
3. 页面验证码或登录失效时触发告警并人工处理。

## 4. 功能范围
### 4.1 MVP 范围
1. 连接市场（应用展示、平台分类、连接引导）。
2. 连接任务（任务创建、参数配置、调度、执行记录、重试）。
3. 账号管理（多平台账号与店铺绑定、验证码转发配置）。
4. 存储管理（飞书多维表、MySQL、钉钉表格）。
5. 消息通知（飞书/钉钉/企微 Webhook）。
6. 管理后台（应用上架、模板配置、运行总览）。

### 4.2 非 MVP（后续）
1. 多租户计费与配额体系。
2. 可视化流程编排（拖拽式）。
3. 平台 SDK/插件生态开放。

## 5. 详细功能设计
### 5.1 连接市场
1. 默认按推荐排序展示连接应用卡片。
2. 支持按一级平台/二级平台筛选（最多两层）。
3. 卡片支持查看详情、发起连接、查看示例字段。
4. 连接流程：选择应用 -> 绑定店铺 -> 配置参数 -> 设置存储 -> 启用任务。

### 5.2 连接任务
1. 任务列表以应用卡片 + 实例数展示，一个应用可绑定多个店铺实例。
2. 支持周期调度（默认每日早 8 点），并支持自定义 Cron。
3. 点击任务可在侧边栏进行参数配置、日志查看、触发执行、失败重试。
4. 支持视图切换：按机器人、按平台、按账号。
5. 支持多条件搜索与过滤（平台、店铺、应用、状态、时间范围）。

### 5.3 账号管理（店铺管理）
1. 支持多平台账号凭据管理：账号、密码、店铺名、平台特有字段。
2. 支持账号与连接应用实例绑定。
3. 支持验证码转发方式配置（如短信中转、人工输入、第三方识别）。
4. 对敏感字段进行加密存储与脱敏展示。

### 5.4 执行记录
1. 展示任务执行历史、状态、耗时、失败原因、重试次数。
2. 支持查看结构化执行日志（步骤级别）。
3. 支持按条件筛选并导出执行记录。

### 5.5 存储管理
1. 飞书多维表：支持 App/Token/表 ID 配置。
2. MySQL（优先）：支持主机、端口、库名、用户名、密码、表映射。
3. 钉钉表格：支持凭据与目标表配置。
4. 配置表单按存储类型动态渲染，仅展示相关参数。

### 5.6 消息通知
1. 支持飞书、钉钉、企微 Webhook。
2. 支持失败通知、重试失败通知、账号失效通知。
3. 支持通知频控与去重策略（避免告警风暴）。

### 5.7 应用上架后台
1. 管理员可新增/下架连接应用。
2. 可配置应用参数模板、字段映射模板、默认调度策略。
3. 具备审核与版本发布能力（记录变更日志）。

## 6. 技术方案
### 6.1 后端技术栈
1. Python 3.12 + FastAPI（API 层）。
2. SQLAlchemy 2.x + Alembic（ORM 与迁移）。
3. Pydantic v2（数据校验）。
4. DrissionPage（浏览器自动化采集）。
5. APScheduler（轻量调度）或 Celery + Redis（分布式任务）。
6. MySQL 8.x（业务主库）+ Redis（缓存、幂等、限流）。
7. Structlog/Loguru + OpenTelemetry（日志与链路追踪）。
8. uv（Python 依赖与环境管理）。

### 6.2 前端技术栈
1. Vue 3 + TypeScript + Vite。
2. Naive UI（基础组件）。
3. Vue Router + Pinia（路由与状态管理）。
4. Axios（请求层）+ ECharts（统计图表）。
5. Vitest + Playwright（单测与端到端测试）。

### 6.3 架构分层
1. Connector Adapter 层：平台页面适配与采集逻辑。
2. Orchestrator 层：任务编排、调度、重试、幂等控制。
3. Data Pipeline 层：清洗、映射、落库。
4. Notification 层：统一消息模板与 webhook 发送。

## 7. 数据模型（核心）
1. `platform`：平台定义。
2. `connector_app`：连接应用定义与版本。
3. `shop_account`：店铺账号与凭据（加密字段）。
4. `task_instance`：任务实例（应用 + 店铺 + 参数 + 调度）。
5. `task_run`：任务运行记录（状态、耗时、错误码）。
6. `storage_config`：存储配置（类型 + 动态参数）。
7. `notification_config`：通知配置与策略。

## 8. 非功能要求
1. 安全：敏感信息加密存储，接口鉴权，操作审计。
2. 稳定性：失败自动重试、熔断、超时控制。
3. 性能：支持并发任务执行与队列削峰。
4. 可观测：任务级指标、告警、日志追踪闭环。

## 9. 里程碑计划（建议）
1. M1（第 1-2 周）：需求冻结、架构评审、基础工程初始化。
2. M2（第 3-5 周）：连接市场、账号管理、任务管理基础能力。
3. M3（第 6-7 周）：执行记录、通知中心、存储管理（MySQL/飞书）。
4. M4（第 8 周）：联调压测、灰度上线、验收。

## 10. 验收标准
1. 可完成“连接应用 -> 绑定店铺 -> 配置任务 -> 自动执行 -> 入库 -> 失败告警”完整闭环。
2. 至少 3 个平台接入通过验收用例。
3. 关键页面和接口具备基础自动化测试。
4. 运维手册与故障处理手册可用。

## 11. 风险与应对
1. 平台页面变更频繁：建立适配器版本化与回归测试机制。
2. 登录风控/验证码：提供人工兜底流程与验证码中转策略。
3. 大促流量峰值：任务分片、队列限流与优先级调度。
4. 数据一致性：幂等键 + 重跑机制 + 对账任务。
